# 补充实验执行说明

这份说明对应以下新增脚本：

- `evaluate_multi_helper_leakage.py`
- `evaluate_public_r_linkability.py`
- `evaluate_iom_baseline.py`
- `evaluate_irreversibility_attack.py`
- 一键运行脚本：`run_additional_experiments.sh`

这些脚本都已经尽量仿照仓库中现有实验脚本的写法，默认使用：

- `MODEL_PATH = checkpoints/final_model.pth`
- `DATA_ROOT = /root/autodl-tmp/FVC2004`
- 独立 `OUTPUT_DIR`

同时支持服务器环境变量覆盖：

- `FVC2004_ROOT`
- `RGSS_MODEL_PATH`

---

## 1. 推荐的整体补实验顺序

建议按下面顺序跑：

### 第一优先级
1. `evaluate_multi_helper_leakage.py`
2. `evaluate_public_r_linkability.py`

原因：这两项最直接回应老师关于 **helper data 累积泄露** 和 **R 明文 side channel** 的质疑。

### 第二优先级
3. `evaluate_iom_baseline.py`

原因：补一个比 BioHashing 更强的 cancelable baseline，有助于增强相关工作和公平比较。

### 第三优先级
4. `evaluate_irreversibility_attack.py`

原因：用于补“不可逆性 / reconstruction attack”这条，和你正文里承认的 helper-data limitation 形成闭环。

## 2. 一键运行方式

### 2.1 默认路径直接运行

如果服务器路径刚好就是：

- 数据：`/root/autodl-tmp/FVC2004`
- 模型：`checkpoints/final_model.pth`

那么直接在项目根目录运行：

```bash
bash run_additional_experiments.sh
```

### 2.2 指定服务器数据和模型路径

如果你的服务器路径不一样，先设置环境变量：

```bash
export FVC2004_ROOT=/你的服务器/FVC2004目录
export RGSS_MODEL_PATH=/你的服务器/checkpoints/final_model.pth
bash run_additional_experiments.sh
```

### 2.3 只跑部分实验

例如只先跑最关键的两项：

```bash
export FVC2004_ROOT=/你的服务器/FVC2004目录
export RGSS_MODEL_PATH=/你的服务器/checkpoints/final_model.pth
RUN_ONLY="multi_helper public_r" bash run_additional_experiments.sh
```

支持的 `RUN_ONLY` 短名：

- `multi_helper`
- `public_r`
- `iom`
- `irreversibility`

## 3.1 `evaluate_multi_helper_leakage.py`

### 目的
回答老师的问题：

> 多次重发模板后 helper data 是否累积泄露？

### 核心实验内容

#### A. Cumulative helper-bit union
- 对同一用户生成多条 revocation / helper record
- 统计随着记录数增加，公开 helper data 累积暴露了多少唯一位置

#### B. Multi-record reliable-position linkability
- 比较同一用户多记录 vs 不同用户多记录的集合相似性
- 观察多记录联合后 linkability 是否增强

#### C. Multi-record helper-aware impostor attack
- 比较：
  - 单条 helper record
  - 多条 helper record 联合观察
- 看攻击者候选选择是否变强

### 输出目录
- `results_multi_helper_leakage/`

### 重点文件
- `cumulative_exposure.png`
- `multi_record_linkability.png`
- `multi_record_attack.png`
- `multi_helper_leakage.json`

### 跑完重点先看什么
1. `cumulative_exposure.png`
   - 看多条 helper 记录下暴露位是否迅速接近整个 1024 维空间
2. `multi_record_linkability.png`
   - 看 union/intersection 后 mated 与 non-mated 是否更可分
3. `multi_record_attack.png`
   - 看 multi-record attack FAR 是否明显高于 single-record

---

## 3.2 `evaluate_public_r_linkability.py`

### 目的
回答老师的问题：

> R 明文 vs R 保护的 linkability 对比

### 核心实验内容

#### A. Public-R linkability
- 直接使用公开 `R` 集合进行 linkability 度量

#### B. Protected-R baseline
- 用一个受保护表示（token/hash 风格）近似模拟 `R` 保护后的状态
- 看 linkability 是否下降

#### C. Helper-template distance linkability
- 再看 helper `h` 自身距离分布是否还保留明显 linkability

### 输出目录
- `results_public_r_linkability/`

### 重点文件
- `public_vs_protected_r_linkability.png`
- `helper_distance_linkability.png`
- `public_r_linkability.json`

### 跑完重点先看什么
1. `public_vs_protected_r_linkability.png`
   - public-R 的 EER 是否明显低于 protected-R
2. `public_r_linkability.json`
   - 看 public 与 protected 两种条件的均值和 EER 对比

## 3.3 `evaluate_iom_baseline.py`

### 目的
回答老师的问题：

> IoM hashing baseline

### 核心实验内容

#### A. ROC comparison
比较：
- StableCTM
- BioHashing
- IoM-style baseline

#### B. GAR–key-length comparison
比较：
- StableCTM + BCH
- BioHashing + BCH
- IoM-style + BCH
- RGSS

### 输出目录
- `results_iom_baseline/`

### 重点文件

- `gs_iom_vs_ctm_rgss.png`
- `iom_baseline_results.json`

### 跑完重点先看什么
1. `gs_iom_vs_ctm_rgss.png`
   - 看 IoM-style 的 `k50` 相对 BioHashing/StableCTM/RGSS 的位置
2. `iom_baseline_results.json`
   - 看 `k50_summary`

## 3.4 `evaluate_irreversibility_attack.py`

### 目的
回答老师的问题：

> 从 helper data 预测原始 deep hash 或 protected template，证明不可逆性

### 核心实验内容

这里不是训练一个复杂攻击模型，而是先做一个**结构性 irreversibility 分析**：

- helper data 能准确恢复什么部分？
- 哪一部分仍不可恢复？
- 剩余不确定性有多大？

脚本会量化：
- full protected template reconstruction accuracy
- reliable subset reconstruction accuracy
- 已知区域和未知区域占比

### 输出目录
- `results_irreversibility_attack/`

### 重点文件
- `reconstruction_accuracy.png`
- `uncertainty_breakdown.png`
- `irreversibility_attack.json`

### 跑完重点先看什么
1. `reconstruction_accuracy.png`
   - 看 helper-only 是否只能在“全模板层面”拿到有限准确率
2. `uncertainty_breakdown.png`
   - 看 unresolved reliable region 还剩多少
3. `irreversibility_attack.json`
   - 看 `unresolved_entropy_upper_bound_bits`

## 7. 当前这批脚本的定位

这批脚本的定位是：

> **先用与你当前项目完全兼容的方式，快速跑出第一轮补充实验结果。**

它们的优势是：
- 不改你原有训练/评估框架
- 输出目录独立清楚
- 方便服务器直接运行
- 方便后续继续微调和补图

如果第一轮结果不错，下一步再做：
- 图形美化
- 更严格的统计显著性
- 主文 / 补充材料排版整理
